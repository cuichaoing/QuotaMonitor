## 第 0 章：测试设计原则与知识体系

### 0.1 测试的 4 个目的

这 95 个测试不是为"覆盖率达标"而写的。它们围绕 4 个目的：

| 目的 | 占比 | 典型用例 |
|------|------|---------|
| **A. 固化踩过的坑** | 30% | KimiCLI User-Agent / GLM 裸 Token / MiniMax 反向字段 |
| **B. 保护核心契约** | 40% | 5h 窗口指纹 / 4 级节流 / 告警去重 / 数据平滑 |
| **C. 文档化行为** | 20% | isRetryable 矩阵 / 阈值分级 / ProviderKind 字段 |
| **D. 回归检测** | 10% | 浮点精度 / 边界值 / 特殊字符 |

**为什么 A 类占比最高？** 因为 QuotaMonitor 涉及的 3 个平台 API 都有反直觉行为。如果不固化这些行为，下一次有人"优化代码"时很可能引入 bug。

### 0.2 测试设计遵循的 5 条原则

#### 原则 1：测试行为而非实现

**反例**（测了实现细节）：
```swift
// 错误：测试内部状态
XCTAssertEqual(quota.raw?.count, 5)
```

**正例**（测用户可见行为）：
```swift
// 正确：测试 API 输出的契约
XCTAssertEqual(window.used, 40)  // 用户能看到 40%
```

应用：所有 95 个测试都断言"对外可见的状态"（返回值、错误类型、计算结果），不依赖私有属性。

#### 原则 2：一个测试一个断言意图

每个测试方法只验证**一个具体的契约**。即使中间有多个 XCTAssert，它们都服务于同一个意图。

**反例**：
```swift
// 错误：混了多意图
func testKimiEverything() {
    // 测了鉴权、解析、错误码、端点...
}
```

**正例**：
```swift
func testBearerAuth_andRequiredUserAgent() {
    // 只测鉴权头
}
```

#### 原则 3：反直觉场景必测

凡是代码里有"非显然"逻辑的地方（注释里有"关键"、"防坑"、"反直觉"），必须有一个测试**显式**覆盖。

**本项目反直觉清单**（已被测试固化）：
- Kimi 必须带 `User-Agent: KimiCLI/1.30.0`，否则命中普通额度而非 Coding Plan
- GLM `Authorization` 头**禁止** "Bearer " 前缀
- MiniMax `current_interval_usage_count` 是"剩余"不是"已用"
- 告警同窗口同 provider 升级才发，降级不发
- 休眠期刷新间隔返回 nil，避免被定时器唤醒
- 5h 窗口指纹 = `floor(epoch/18000)`

#### 原则 4：测试代码本身要可读

测试是给"未来的自己"和"接手的人"读的。每个测试都配 1-3 行注释说明意图。

#### 原则 5：覆盖率不是目标

我们不追求"100% line coverage"，但追求"100% 关键契约覆盖"。本次 95 个测试覆盖了：
- 8 个反直觉场景（100%）
- 4 个核心契约（100%）
- 5 个错误码路径（100%）
- 3 个平台 Provider（100%）

剩余未覆盖的代码是：
- UI 渲染层（SwiftUI 测试成本高，改用截图回归）
- 系统集成层（Keychain / UN 通知，需要真机 + 权限）
- main 入口（@main 没有测试价值）

### 0.3 知识体系：3 个来源

这些测试的设计不是凭空而来，来源于 3 个知识输入：

#### 输入 1：3 平台 API 调研报告（设计期输入）

调研报告指出每个平台有"反直觉"字段和"鉴权陷阱"：

| 平台 | 反直觉点 | 应对 |
|------|---------|------|
| Kimi | User-Agent 决定额度类型 | KimiProvider 强制带 `KimiCLI/1.30.0` |
| MiniMax | 字段名 `usage` 但语义是 remaining | MiniMaxProvider 反算法 |
| GLM | Authorization 头语义特殊 | GLMProvider 用裸 Token |

每个反直觉点都对应一个测试。

#### 输入 2：QuotaCat 1.0.12 历史经验（来自 1.x 项目）

详见 `docs/legacy/DECISIONS.md`：
- 7 个吸收的设计模式（actor 并行、Keychain 存储、指纹去重等）
- 8 个反模式（写过的 bug）
  - 串行抓取（→ v0.1 强制并行 + §9.1 测试）
  - 不带 User-Agent 被 403（→ §6.3 测试）
  - Bearer 前缀加错被 401（→ §8.1 测试）
  - 字段语义理解反（→ §7.1 测试）
  - 通知刷屏（→ §12.9-§12.10 去重测试）
  - used 倒退导致 UI 跳（→ §9.4 平滑测试）
  - 休眠期被定时器唤醒（→ §11.18 测试）
  - 阈值不明确（→ §3.1-§3.10 错误矩阵测试）

#### 输入 3：Swift 测试技术栈（实现期输入）

为了让测试**真的能跑**且**稳定**，需要解决 4 个 Swift 特有的工程问题：

| 问题 | 解法 | 对应测试 |
|------|------|---------|
| URLSession 不走 MockURLProtocol | URLProtocol 拦截 + Task.detached | §6.3, §7.4, §13.1 |
| 异步闭包中修改 var 报错 | 用 `actor TickCounter` 包装 | §10.6 |
| `try?` 推断不出 Double? vs Double | 用 `XCTUnwrap(primaryWindow)` | §2.1-§2.6 |
| UserDefaults 污染真机 | 每个测试用独立 `UserDefaults(suiteName:)` | §11.14-§11.16 |

### 0.4 测试目录的设计

13 个测试文件按"职责层"切分，对应生产代码的分层：

```
生产代码分层              测试文件              测什么
─────────────────────────────────────────────────────
Core/Models/        →   ProviderKindTests        枚举字段
                    →   QuotaWindowTests         派生计算
                    →   ProviderQuotaTests       5h 指纹
Core/Errors/        →   QuotaErrorTests          错误矩阵
Core/Protocols/     →   KeychainStoreTests       InMemory mock
Services/Networking/→   KimiProviderTests        鉴权 + 解析
                    →   MiniMaxProviderTests
                    →   GLMProviderTests
                    →   NetworkingServiceTests   并行 + 平滑
Services/Throttling/→   SmartRefreshSchedulerTests 4 级节流
State/              →   SettingsStoreTests       Keychain + UD
UI/Notifications/   →   AlertStateMachineTests   状态机
                    →   BarkClientTests          URL + 编码
```

**为什么按层分而不是按"功能"分？** 当某层代码改动时，定位测试文件最快。"网络层出 bug" → 看 `NetworkingServiceTests` / `*ProviderTests`。

### 0.5 一个测试方法的 3 段描述

下面每条测试都按以下结构描述：

1. **是什么**：测试名 + 一句话定义（这个测试的标题）
2. **内容**：具体测什么（输入数据 + 断言点）
3. **作用**：为什么需要这个测试（保护什么 / 防什么 bug / 文档化什么行为）

## 第 1 章：ProviderKindTests (7)

**测试层**：Core/Models
**生产代码**：`QuotaMonitor/Core/Models/ProviderKind.swift`
**标的**：`ProviderKind` 枚举（Kimi / MiniMax / GLM）的字段稳定性 —— `displayName` / `defaultBaseURL` / `requiredUserAgent` / `requiresBearerPrefix` / `sortOrder`

### 1.1 `testAllCases_count`

- **是什么**：枚举 case 数量测试
- **内容**：断言 `ProviderKind.allCases.count == 3`
- **作用**：防止有人误删某个 case。如果未来加第 4 个平台（如 DeepSeek），这个测试会失败提醒"需要更新测试"。

### 1.2 `testSortOrder_unique`

- **是什么**：sortOrder 唯一性测试
- **内容**：3 个 case 的 sortOrder 放进 `Set`，断言 Set 大小 = 3
- **作用**：sortOrder 决定 UI 排列顺序和 `enabledKinds()` 返回顺序。如果两个平台有相同 sortOrder，UI 会出现错乱（位置不稳定）。这个测试保证 sortOrder 严格唯一。

### 1.3 `testDisplayNames_unique`

- **是什么**：displayName 唯一性测试
- **内容**：3 个 case 的 displayName 放进 `Set`，断言 Set 大小 = 3
- **作用**：displayName 直接显示在 UI 上（Popup 卡片标题、状态栏 tooltip）。如果有重复，用户在设置面板里分不清是哪个平台。

### 1.4 `testUserAgent_OnlyKimiRequired`

- **是什么**：User-Agent 需求矩阵测试
- **内容**：断言 Kimi 的 `requiredUserAgent` 非 nil，MiniMax 和 GLM 都 nil
- **作用**：固化"只有 Kimi 需要 User-Agent"的设计决策。如果误给 GLM 加了 User-Agent，请求会带无效头；如果误删 Kimi 的 User-Agent，所有 Kimi 请求会命中普通额度而非 Coding Plan（这是真金白银的损失）。

### 1.5 `testBearerPrefix_GLMForbidden`

- **是什么**：【关键防坑】GLM 鉴权头前缀测试
- **内容**：断言 `requiresBearerPrefix` 在 3 平台的值
- **作用**：这是项目里第一个被 fix 过的 bug 类型。GLM 的鉴权头必须裸 Token，加 "Bearer " 前缀会被智谱服务端 401。这个测试**显式**记录"GLM 永远 false"。

### 1.6 `testBaseURLs_allValid`

- **是什么**：3 平台 base URL 合法性测试
- **内容**：遍历 allCases，断言 host 非 nil + scheme 以 "https" 开头
- **作用**：保证所有 API 端点都是 HTTPS（明文 Key 不能走 HTTP）。如果有人误把 URL 改成 `http://...`，会立刻失败。

### 1.7 `testKimiUserAgent_format`

- **是什么**：Kimi User-Agent 内容格式测试
- **内容**：断言 `requiredUserAgent?.contains("KimiCLI")`
- **作用**：Kimi 服务端按 User-Agent 路由不同额度池。如果格式改了（比如去掉 KimiCLI），Coding Plan 专属额度会失效。这个测试锁住"必须含 KimiCLI 标识"。

## 第 2 章：QuotaWindowTests (6)

**测试层**：Core/Models
**生产代码**：`QuotaMonitor/Core/Models/QuotaWindow.swift`
**标的**：`QuotaWindow` 模型的派生计算 —— `usedPercent` / `remainingPercent` / `remaining`

### 2.1 `testUsedPercent_basic`

- **是什么**：基础百分比计算测试
- **内容**：`QuotaWindow(total: 100, used: 30, ...)`，断言 usedPercent/remainingPercent/remaining
- **作用**：这是所有 UI 数字的源头。如果 `usedPercent` 算错，整个状态栏和 Popup 都错。

### 2.2 `testUsedPercent_cappedAt100`

- **是什么**：usedPercent 100% 封顶测试
- **内容**：`QuotaWindow(total: 100, used: 150, ...)`，断言 usedPercent = 100
- **作用**：服务端可能因为时钟不同步返回 used > total（极少见但发生过）。如果 usedPercent 可以 > 100，UI 进度条会绘制溢出。

### 2.3 `testUsedPercent_zeroTotal_returnsZero`

- **是什么**：除零保护测试
- **内容**：`QuotaWindow(total: 0, used: 0, ...)`，断言 usedPercent = 0
- **作用**：total=0 时除法会崩溃。必须显式 return 0。这条测试防止有人"优化"代码时去掉除零保护。

### 2.4 `testRemaining_neverNegative`

- **是什么**：remaining 永不为负测试
- **内容**：`QuotaWindow(total: 100, used: 150, ...)`，断言 remaining = 0
- **作用**：UI 显示"剩余 XX"如果为负会很奇怪。这个测试保证 `remaining = max(0, total - used)`。

### 2.5 `testUsedPercent_GLMDirectPercent`

- **是什么**：GLM 直返百分比测试
- **内容**：`QuotaWindow(total: 100, used: 75, ...)`，断言 usedPercent = 75
- **作用**：GLM 端点直接返回百分比（不像 Kimi 那样返回 limit/remaining 让客户端算）。这个测试固化"GLM 模式下 used 直接等于百分比"。

### 2.6 `testEquatable`

- **是什么**：Equatable 一致性测试
- **内容**：构造 2 个完全相同的 + 1 个 used=40 的 QuotaWindow
- **作用**：`==` 用于 NetworkingService.applySmoothing 里比较"是不是同一个窗口"。如果 Equatable 行为反直觉（比如按 reference 比较），数据平滑会失效。

## 第 3 章：QuotaErrorTests (11)

**测试层**：Core/Errors
**生产代码**：`QuotaMonitor/Core/Errors/QuotaError.swift`
**标的**：`QuotaError` 枚举的 `isRetryable` / `isUserActionable` / `errorDescription` 分类

### 3.1 `testRateLimited_isRetryable`

- **是什么**：限流可重试测试
- **内容**：断言 429 限流错误（含/不含 retryAfter）都 isRetryable
- **作用**：限流是临时性错误，重试（按 retryAfter 等待）有效。这个测试保证重试策略会处理这种情况。

### 3.2 `testNetworkUnreachable_isRetryable`

- **是什么**：网络错可重试测试
- **内容**：断言 `networkUnreachable(.kimi).isRetryable == true`
- **作用**：Mac 偶尔断网（WiFi 切换等）属临时性，下一次刷新大概率能成功。重试应恢复。

### 3.3 `testServerError5xx_isRetryable`

- **是什么**：5xx 可重试测试
- **内容**：500/502/503 三个状态码都断言 isRetryable
- **作用**：5xx 是服务端问题（过载、维护），客户端重试可能有效。如果 5xx 不可重试，临时性的服务端问题会导致永久失败。

### 3.4 `testServerError4xx_notRetryable`

- **是什么**：4xx 不可重试测试
- **内容**：400/404 都断言 !isRetryable
- **作用**：4xx 是客户端请求问题（参数错、URL 错），重试同样会失败。错误标记为不可重试避免浪费请求。

### 3.5 `testKeyInvalid_notRetryable`

- **是什么**：Key 无效不重试测试
- **内容**：断言 `keyInvalid(.kimi).isRetryable == false`
- **作用**：Key 错了（过期、被撤销），重试也无效。必须让用户去更新 Key，不能无意义刷请求。

### 3.6 `testKeyMissing_notRetryable`

- **是什么**：Key 缺失不重试测试
- **内容**：断言 `keyMissing(.kimi).isRetryable == false`
- **作用**：用户还没填 Key，重试当然没用。这个测试 + `isUserActionable` 配合，让 UI 引导用户去设置。

### 3.7 `testCancelled_notRetryable`

- **是什么**：取消不重试测试
- **内容**：断言 `QuotaError.cancelled.isRetryable == false`
- **作用**：用户主动取消或系统取消，重试违背用户意图。

### 3.8 `testKeyMissing_isUserActionable`

- **是什么**：【关键】Key 缺失引导测试
- **内容**：3 个平台的 `keyMissing` 都断言 isUserActionable
- **作用**：这个属性直接决定 UI 是否显示"去设置"按钮。如果 false，用户看到"Key 缺失"红色错误但不知道怎么处理，会很迷茫。

### 3.9 `testKeyInvalid_isUserActionable`

- **是什么**：Key 无效提示测试
- **内容**：断言 `keyInvalid(.kimi).isUserActionable == true`
- **作用**：Key 过期/撤销时，UI 应主动告诉用户"请检查 Key 是否过期"，而不是只显示一个干巴巴的错误。

### 3.10 `testNetworkErrors_notUserActionable`

- **是什么**：网络错不引导测试
- **内容**：`networkUnreachable` 和 `rateLimited` 都断言 !isUserActionable
- **作用**：网络问题用户什么都做不了（不是他能修复的），不应显示"去设置"按钮误导。

### 3.11 `testErrorDescription_containsProviderName`

- **是什么**：错误描述可读性测试
- **内容**：断言 `keyMissing(.kimi).errorDescription?.contains("Kimi")`
- **作用**：错误描述字符串会显示在系统通知和 Popup 里。必须含平台名，否则用户不知道是哪个平台出的问题。

## 第 4 章：KeychainStoreTests (6)

**测试层**：Core/Protocols（用 InMemoryKeychainStore 测，行为与生产 KeychainStore 一致）
**生产代码**：`QuotaMonitor/Services/Security/KeychainStore.swift` + `InMemoryKeychainStore`
**标的**：Keychain CRUD 行为正确性

### 4.1 `testSaveReadDelete`

- **是什么**：基础 CRUD 测试
- **内容**：未存 → save → exists → read → delete → 不存在，6 步完整流程
- **作用**：所有 Keychain 操作的"最常用路径"。如果这步出错，API Key 永远存不下来。

### 4.2 `testDeleteNonExistent_doesNotThrow`

- **是什么**：删不存在不抛错测试
- **内容**：新 store 上直接 delete
- **作用**：用户在设置面板多次点"删除"，第二次时 Key 已经不存在。如果 delete 抛错，UI 闪退。`KeychainStore` 用 `SecItemDelete` 在 errSecItemNotFound 时也视为成功（不抛错）。

### 4.3 `testMultipleProviders_isolated`

- **是什么**：3 平台 Key 隔离测试
- **内容**：分别 save 3 个 key + 删一个 + 检查另两个未受影响
- **作用**：3 个平台的 Key 用 `provider.rawValue` 作 account 区分。这个测试保证不会因为 key 拼写错导致一个平台的 Key 覆盖另一个。

### 4.4 `testOverwriteKey`

- **是什么**：覆盖而非追加测试
- **内容**：连续 save v1/v2，读出应为 v2
- **作用**：`SecItemUpdate` 在已存在时覆盖。这个测试防止有人"简化代码"为每次都 delete+add（多一次系统调用，慢）。

### 4.5 `testEmptyKey_allowed`

- **是什么**：空字符串允许存储测试
- **内容**：save `""` 能存能读
- **作用**：Keychain 层不校验业务规则（"Key 不能为空"由 SettingsStore 负责）。这个测试划清"Keychain 只管存储"和"业务校验"的边界。

### 4.6 `testSpecialCharactersInKey`

- **是什么**：特殊字符存储测试
- **内容**：save 含 `!@#$%^&*()_+-={}[]|:;"'<>,.?/` 的 key
- **作用**：Keychain 内部用 Data 存储（UTF-8 编码），理论上支持所有字节。这个测试防止有人误改实现为用 `String` 拼接。

## 第 5 章：ProviderQuotaTests (3)

**测试层**：Core/Models
**生产代码**：`QuotaMonitor/Core/Models/ProviderQuota.swift`
**标的**：`ProviderQuota.windowFingerprint` —— 5h 窗口指纹（告警去重的核心）

### 5.1 `testWindowFingerprint_stableWithin5Hours`

- **是什么**：5h 窗口内指纹稳定测试
- **内容**：3 个时间戳（基准/+60s/+3600s），断言 fingerprint 相等
- **作用**：windowFingerprint 是告警去重的 key。如果同一窗口内 fingerprint 变化，告警会重复发（用户被刷屏）。

- **内容）：同窗口 80% → 92%
- **作用）：75% warning → 90% critical 是严重化，必须再发。如果 fail，用户会错过"危险升级"信号。

### 12.10 `testEvaluate_criticalToWarning_doesNotEmit`

- **是什么**：【关键】等级降级不重发测试
- **内容）：同窗口 92% → 80%
- **作用）：90% critical → 80% warning 是缓解，不应再发。否则用户在半小时内会被通知两次（先 critical，后 warning），很烦。

### 12.11 `testEvaluate_multiplePlatforms_independentDedup`

- **是什么）：多平台独立测试
- **内容）：Kimi + GLM 都 80%
- **作用）：每个平台的去重状态独立。如果 fail，Kimi 告警会拦截 GLM 告警。

### 12.12 `testReset_clearsDedupState`

- **是什么）：reset 重置测试
- **内容）：发一次 → reset → 再发
- **作用）：reset() 用于单元测试清理 + 调试时强制重发。生产代码不应该调它。

## 第 13 章：BarkClientTests (4)

**测试层**：UI/Notifications
**生产代码**：`QuotaMonitor/UI/Notifications/BarkClient.swift`
**标的**：URL 构造 + 编码 + 错误处理

### 13.1 `testPush_buildsCorrectURL`

- **是什么）：URL 构造测试
- **内容）：调 push，拦截检查 req.url
- **作用）：Bark 协议规定 URL 格式 `{server}/{deviceKey}/{title}/{body}`。如果 URL 拼错，Bark 服务端会返回 4xx，iPhone 收不到推送。

### 13.2 `testPush_dangerLevel_addsAlarmSound`

- **是什么**：【行为】danger 等级 alarm 音测试
- **内容）：level=.danger push → 检查 query 含 sound=alarm
- **作用）：danger 等级（99%）时 iPhone 应有刺耳的 alarm 音提醒。其他等级是默认提示音（不吵）。

### 13.3 `testPush_4xx_throwsError`

- **是什么）：4xx 错误处理测试
- **内容）：mock 400，断言抛 serverError
- **作用）：Bark 服务端 4xx（device key 错、签名错）应让调用方知道。如果静默忽略，用户以为推送成功实际没收到。

### 13.4 `testPush_urlEncoding_handlesSpecialChars`

- **是什么）：中文/空格编码测试
- **内容）：push 含中文 title，断言解码后能拿到原文
- **作用）：Bark URL 直接拼到 path 部分，必须 percent-encode 中文/空格。否则服务端解析失败。
