## 4. 法律与社区文件（push 前必做）

### 4.1 LICENSE（缺失 — 优先级最高）
- README 写了 "MIT" 但仓库根**没有** LICENSE 文件
- 必须创建 `LICENSE`（MIT 全文本，非模板）
- 推荐协议对比：

| 协议 | 特点 | 适合 |
|------|------|------|
| **MIT** | 最宽松，几乎无限制 | 工具类、闭源友好 |
| **Apache 2.0** | 宽松 + 专利授权 + 商标保护 | 企业级项目 |
| **GPL v3** | 强 copyleft，派生作品必须开源 | 教学/学术 |
| **BSD 3-Clause** | 宽松 + 禁止用作者名义背书 | 类 Unix 工具 |

**推荐 QuotaMonitor 用 MIT**（与 README 一致 + 工具类惯例）。

### 4.2 社区文件
| 文件 | 必要性 | 内容 |
|------|--------|------|
| `CONTRIBUTING.md` | 中 | 如何提 Issue / PR / 跑测试 |
| `CODE_OF_CONDUCT.md` | 低 | Contributor Covenant v2.1（标准模板） |
| `SECURITY.md` | 中 | 安全漏洞报告流程（私有邮箱，不用 GitHub Issue） |
| `.github/ISSUE_TEMPLATE/bug_report.md` | 中 | 必填项：版本/系统/API Key 状态/log |

## 9. 风险与建议

### 9.1 风险
| 风险 | 概率 | 缓解 |
|------|------|------|
| 误传真实 API Key | 中 | push 前再 grep 一次 |
| 反编译暴露用户 Key（已存本地） | 低 | Keychain 加密 + `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` 已做 |
| GLM/MiniMax 端点变更（v0.x 教训） | 高 | README 写明"API 字段可能变化，遇 bug 提 Issue" |
| 用户反馈噪音（Key 配错等） | 高 | 写详细 troubleshooting + Issue 模板强制填版本 |
| 收到 DMCA / 商标投诉 | 极低 | 商标：项目名 QuotaMonitor + 三大平台名是描述性使用 |

### 9.2 长期建议
- **首个 release 后**：建立 CHANGELOG 节奏（功能/修复/破坏性变更三段式）
- **每月 1 号**：跑一次全平台 API 字段扫描，警惕变化
- **建立 issue 标签体系**：`bug` / `enhancement` / `docs` / `platform:kimi|minimax|glm`
