# QuotaMonitor v1.0 - 从编译到日常使用完整步骤报告

**目标读者**：刚 clone 完代码、准备第一次跑起来的用户
**覆盖范围**：从打开终端 → 状态栏看到实时数字 → 收到第一次告警 → 可选配 Bark 推送
**总耗时**：约 15-20 分钟（含等待 API Key + 配置）

---

| 步骤 | 动作 | 工具 | 耗时 |
|------|------|------|------|
| 1 | 验证编译环境 | 终端 | 1 分钟 |
| 2 | 编译 release 版本 | 终端 | 2-3 分钟 |

| 9 | （可选）配置 Bark iOS 推送 | iPhone + 鼠标 | 3 分钟 |
| 10 | 验证节流 + 退出 | 终端 | 1 分钟 |

## 步骤 1：验证编译环境

### 1.1 检查 Xcode Command Line Tools

```bash
xcode-select -p
```

**预期输出**（任一即可）：
- `/Applications/Xcode.app/Contents/Developer`
- `/Library/Developer/CommandLineTools`

**如果不是**：
```bash
xcode-select --install
# 弹出系统对话框，按提示安装（约 100MB）
```

### 1.2 检查 Swift 版本

```bash
swift --version
```

**预期输出**（参考）：
```
Apple Swift version 5.9+ (swiftlang-...)
```

如果是 Swift 5.7 或更早，编译可能失败。需要升级 Xcode 到 14.0+。

### 1.3 切换到项目目录

```bash
cd <PROJECT_ROOT>
ls
```

**预期输出**（看到这些文件/目录）：
```
CHANGELOG.md
Package.swift
QuotaMonitor/
QuotaMonitorTests/
README.md
USER_GUIDE.md
docs/
```

### 1.4 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| `xcode-select: error: unable to get active developer directory` | 未装 CLT | 跑 `xcode-select --install` |
| `swift: command not found` | PATH 缺 Swift | 装 Xcode 14.0+ |
| 目录不存在 | 路径输错 | 检查 `pwd` 后重新 `cd` |

## 步骤 2：编译 release 版本

### 2.1 跑测试先（推荐）

```bash
swift test
```

**预期输出**（最后几行）：
```
Test Suite 'All tests' passed
Executed 95 tests, with 0 failures (0 unexpected) in 0.471 (0.480) seconds
```

**说明**：如果 95 个测试都过，编译产物是可信的，可以直接编译 release。

### 2.2 编译 release

```bash
swift build -c release
```

**预期输出**（最后几行）：
```
Build complete! (10-30s)
```

**产物位置**：
```
.build/release/QuotaMonitor
```

**说明**：
- 首次编译 2-3 分钟（编译 48 个 .swift 文件）
- 后续编译 < 10 秒（增量）

### 2.3 验证编译产物

```bash
ls -la .build/release/QuotaMonitor
file .build/release/QuotaMonitor
```

**预期输出**：
- 文件存在，大小约 1-2 MB
- 类型：`Mach-O 64-bit executable arm64`

### 2.4 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| `error: unable to find module dependency: 'SwiftUI'` | macOS 版本 < 12 | 升级到 macOS Monterey (12) 或更新 |
| `error: no such module 'UserNotifications'` | SDK 缺 | 装 Xcode 14.0+ |
| `error: the package 'swift-tools-version' ...` | Swift 版本低 | 升级 Xcode |
| 链接错误 | 缓存损坏 | `rm -rf .build && swift build -c release` |

## 步骤 3：启动应用

### 3.1 启动

```bash
.build/release/QuotaMonitor
```

**预期现象**（按时间顺序）：
1. 终端无输出（应用驻留后台）
2. macOS 顶部状态栏（右上角菜单栏）出现一个新图标：柱状图样式
3. 约 1-2 秒后，应用准备就绪

**图标位置**：
```
+-------------------------------------------------------+
| [Wi-Fi] [Battery] [Spotlight] ... [QuotaMonitor图标] [Time] |
+-------------------------------------------------------+
```

### 3.2 检查进程

在另一个终端窗口：
```bash
ps aux | grep QuotaMonitor | grep -v grep
```

**预期**：看到一个进程，路径是 `.build/release/QuotaMonitor`

### 3.3 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| 终端没输出就退出 | 编译产物缺 | 重跑 `swift build -c release` |
| 图标没出现 | 应用没启动 | 看 `Console.app` 找错误日志 |
| 启动后立即退出 | 权限/签名问题 | 尝试 `codesign --force --deep -s - .build/release/QuotaMonitor` |
| 图标在状态栏但点了没反应 | 罕见 bug | 退出重开 |

## 步骤 4：打开设置面板

### 4.1 点击状态栏图标

**方法 A**：左键单击
**方法 B**：右键 → 设置

**预期**：弹出 Popup 窗口（约 360x480），显示：
- 标题 "QuotaMonitor"
- 黄色提示框 "首次使用 - 点击设置填入至少一个平台的 API Key"
- 三个空白的平台卡片（Kimi / MiniMax / GLM）
- 底部 "设置" 按钮

### 4.2 打开设置

在 Popup 左下角点 **"设置"** 按钮。

**预期**：弹出设置窗口（约 520x700），分 4 个区：
1. 平台配置（3 个平台的 Key 输入框 + 启用 checkbox）
2. 颜色阈值（3 个 slider：警告/严重/危险）
3. Bark 推送（可折叠，先不展开）
4. 高级（可折叠，先不展开）

### 4.3 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| 点击状态栏图标无反应 | 应用未启动 | 重跑步骤 3 |
| 设置窗口打不开 | 系统权限 | 系统设置 → 隐私与安全 → 辅助功能 |
| Popup 太小看不见 | 屏幕分辨率 | 改用右键菜单的"设置" |

## 步骤 5：获取并填入 3 个 API Key

### 5.1 获取 Kimi Code Key

1. 浏览器打开 https://kimi.com
2. 登录你的账号
3. 进入"设置"或"开发者"
4. 创建新的 API Key（标记为 "Coding Plan" 类型）
5. 复制 Key（形如 `sk-kimi-abc...`）

**注意**：Kimi 的 Coding Plan Key 和普通 Chat Key 是分开的，**必须**是 Coding Plan。

### 5.2 获取 MiniMax Key

1. 浏览器打开 https://platform.minimax.io
2. 登录
3. 进入"订阅"或"Coding Plan"
4. 找到 "API Key" 部分
5. 复制 Key

**注意**：MiniMax 必须用 **Coding Plan 专属 Key**，不是普通 Chat Key。

### 5.3 获取智谱 GLM Key

1. 浏览器打开 https://bigmodel.cn
2. 登录
3. 进入"API Keys"
4. 创建新 Key
5. 复制 Key

**注意**：GLM 的 Key 不需要加 "Bearer " 前缀（应用会自动处理）。

### 5.4 在 QuotaMonitor 设置中填入

按以下顺序操作：
1. 在 Kimi Code 那一行：
   - 确保左边 checkbox 勾选（默认勾选）
   - 粘贴 Key 到输入框
   - 点击"保存"按钮
   - 出现绿色"已配置"标签

2. 对 MiniMax Coding 重复

3. 对智谱 GLM 重复

**预期**：3 行都出现"已配置"绿色标签。

### 5.5 关闭设置窗口

点窗口右上角关闭按钮，或点底部"完成"按钮。

**预期**：状态栏数字开始变化（应用会立即拉一次新数据）。

### 5.6 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| "保存失败" 提示 | Key 含特殊字符 | 重新复制一次，避免换行/空格 |
| 保存后立即消失 | Keychain 权限问题 | 系统设置 → 钥匙串访问 → 找到 "com.user.quotamonitor.apikeys" |
| 平台一直 401 | Key 类型错 | 重新生成 Coding Plan 专用 Key |

## 步骤 6：允许系统通知

### 6.1 等待通知授权弹窗

应用启动时**会自动**请求通知权限。**注意**：这个弹窗只在**第一次启动**时出现。

如果错过了：
```bash
# 退出应用
# 系统设置 → 通知 → QuotaMonitor → 允许通知
# 重启应用
```

### 6.2 弹窗出现

**预期弹窗**：
```
[QuotaMonitor] 想给您发送通知
[不允许] [允许]
```

**操作**：点 **"允许"**

### 6.3 验证

在 Popup 中如果数据已经达到阈值，会看到右上角系统通知横幅。

或者手动触发测试（步骤 8）。

### 6.4 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| 弹窗没出现 | 应用已跑过，被拒绝过 | 手动去系统设置开启 |
| 通知被静默 | 通知中心"勿扰" | 关掉勿扰模式 |
| 通知没声音 | 系统静音 | 检查系统音量 |

## 步骤 7：验证状态栏数字

### 7.1 等待首次抓取

应用配置完成后，会在 30 秒内（默认 active 间隔）自动抓取一次。

**预期状态栏变化**：
```
启动时：  [图标] ...
配置后：  [图标] K 35 M 60 G 12
```

格式说明：
- `K` = Kimi, `M` = MiniMax, `G` = GLM
- 后面数字 = 已用百分比
- `--` = 该平台未抓到（Key 错/服务端错误）

### 7.2 颜色变化

| 颜色 | 触发条件 |
|------|---------|
| 绿色 | 3 平台都 < 75% |
| 黄色 | 任一平台 75-90% |
| 橙色 | 任一平台 90-99% |
| 红色 | 任一平台 ≥ 99% |

**预期**：根据你实际的配额情况显示对应颜色。

### 7.3 点击状态栏看详细数据

左键点击状态栏图标 → 弹出 Popup。

**预期 Popup 内容**：
- 3 个平台卡片，每个显示：
  - 平台名
  - 百分比（数字 + 进度条）
  - 重置倒计时（如 `重置于 03:42:15`）
- 底部："刷新于 HH:MM:SS" 时间戳

### 7.4 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| 状态栏只有 `--` | Key 错 或 服务端错 | 看 Popup 平台卡片的红色错误提示 |
| 状态栏图标不变 | 应用卡住 | 右键 → 手动刷新 |
| 进度条一直是空 | 解析失败 | 看 docs 里的故障排查 |

## 步骤 8：手动验证告警（推荐）

**目的**：确认告警系统端到端工作正常。

### 8.1 临时调低阈值

打开设置 → 颜色阈值：
- 黄色警告：从 75 拖到 1
- 橙色严重：从 90 拖到 1
- 红色危险：从 99 拖到 1

### 8.2 触发刷新

关闭设置窗口 → 等待 30 秒（自动刷新）或右键 → 手动刷新。

### 8.3 验证告警

**预期**：
- 弹系统通知（macOS 顶部 / 通知中心）
- 标题：`QuotaMonitor - Kimi 配额告警`
- 内容：`Kimi 已用 X%，请注意配额消耗`

**或者**：
- iPhone Bark App 收到推送（如果已配）

### 8.4 恢复默认阈值

测试完后，把 3 个阈值改回 75 / 90 / 99。

### 8.5 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| 没弹通知 | 通知权限被拒 | 步骤 6 重新授权 |
| 通知太多 | 阈值太低 | 调高阈值 |
| Bark 没收到 | 没配/没启用 | 步骤 9 |

## 步骤 9：（可选）配置 Bark iOS 推送

**前置**：iPhone 装了 Bark App（App Store 搜"Bark"，免费）

### 9.1 iPhone 端

1. 打开 Bark App
2. 顶部会显示一个 device key（形如 `abc123def456...`，长字符串）
3. 长按或点击复制

### 9.2 macOS 端

1. QuotaMonitor Popup → 设置
2. 找到"Bark 推送"区
3. 勾选启用
4. 在"Device Key"输入框粘贴 iPhone 的 key
5. 点"保存"

**预期**：下面出现绿色"已配置"标签。

### 9.3 测试推送

1. 重复步骤 8（调低阈值 → 触发刷新）
2. iPhone Bark App 收到通知

### 9.4 失败处理

| 错误 | 原因 | 修法 |
|------|------|------|
| iPhone 没收到 | 设备未联网 | 检查 WiFi |
| device key 错 | 复制不完整 | 重新复制 |
| 自建 Bark 服务器 | 改"服务器"字段 | 填你的 URL |

## 步骤 10：日常使用

### 10.1 启动应用

```bash
# 方式 A：每次手动启动
.build/release/QuotaMonitor

# 方式 B：设为开机启动（推荐）
# 系统设置 → 通用 → 登录项 → 添加 .build/release/QuotaMonitor
```

### 10.2 查看配额

左键点击状态栏图标 → 看 Popup

### 10.3 手动刷新

- Popup 右上角 ⟳ 图标
- 或状态栏右键 → 手动刷新

### 10.4 退出应用

- 状态栏右键 → 退出
- 或 Popup 左下角"退出"
- 或 ⌘Q（Popup 内时）

### 10.5 智能节流

| 状态 | 抓取间隔 | 触发条件 |
|------|---------|---------|
| 活跃 | 30s | VSCode/Xcode/Cursor 等 IDE 前台 + 鼠标活跃 |
| 普通 | 60s | 普通前台应用 |
| 空闲 | 5m | 长时间无操作 |
| 休眠 | 暂停 | Mac 睡眠（系统自动暂停，唤醒后恢复） |

**无需手动配置**，应用自动适配。

### 10.6 调阈值

设置 → 颜色阈值 → 拖动 slider

**推荐设置**：
- 保守型（不常被打扰）：75 / 90 / 99
- 敏感型（频繁提醒）：60 / 80 / 95

### 10.7 暂停某平台

设置 → 取消勾选某平台

**适用场景**：
- 某平台 Key 失效但暂时不想修
- 某平台配额已用完但不想看
- 临时减少网络请求

## 卸载

```bash
# 1. 退出应用
# 状态栏右键 → 退出

# 2. 删除应用文件
rm <PROJECT_ROOT>/.build/release/QuotaMonitor

# 3. 清理 Keychain 里的 API Key
open /Applications/Utilities/Keychain\ Access.app
# 搜索 "quotamonitor" → 删除所有相关条目

# 4. 清理 UserDefaults
defaults delete com.user.quotamonitor 2>/dev/null
# 清理登录项（如果加了）
# 系统设置 → 通用 → 登录项 → 移除 QuotaMonitor
```

## 常见问题速查

| 问题 | 答案 |
|------|------|
| 状态栏数字不更新？ | 右键 → 手动刷新；或检查 Key 是否过期 |
| 通知不弹？ | 系统设置 → 通知 → QuotaMonitor → 允许 |
| Bark 收不到？ | 检查 iPhone 网络 + device key 复制完整 |
| 应用占内存？ | < 30MB（actor + SwiftUI 极轻量） |
| 跟其他 App 冲突？ | 不会，纯独立进程 |
| 隐私？ | API Key 在 macOS Keychain，应用不联网收集数据 |
| 后台运行？ | 是，状态栏常驻，CPU 0%（只在 active 时短暂使用） |
| MacBook 没电能用吗？ | 能，休眠期不抓取，唤醒后继续 |
| 多用户？ | 每个用户独立 Keychain，应用数据不共享 |
