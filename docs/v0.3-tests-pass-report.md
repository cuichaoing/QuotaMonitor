## 二、核心断言亮点

### 1. 反直觉字段（MiniMax）
```swift
// current_interval_usage_count 实际是"剩余"，不是"已用"
XCTAssertEqual(window.used, 30, accuracy: 0.001)         // total=100, remaining=70
XCTAssertEqual(window.remaining, 70, accuracy: 0.001)
```

### 2. 鉴权头防坑（GLM）
```swift
XCTAssertEqual(capturedAuth, "my-raw-glm-token",
               "GLM Authorization 必须是裸 Token")
XCTAssertFalse(capturedAuth?.hasPrefix("Bearer ") ?? false,
               "绝不能加 Bearer 前缀，否则智谱服务端 401")
```

### 3. 必备头（Kimi）
```swift
XCTAssertEqual(req.value(forHTTPHeaderField: "User-Agent"), "KimiCLI/1.30.0",
               "User-Agent 必须带 KimiCLI 标识才能命中 Coding Plan 专属额度")
```

### 4. 数据平滑
```swift
// 第一次 used=60, 第二次 used=40（倒退）
XCTAssertEqual(w2.used, 60, accuracy: 0.001,
               "同窗口内 used 不应倒退，平滑保留上一次的 60")
```

### 5. 并行抓取
```swift
// 3 平台 × 0.2s 延迟，串行 = 0.6s，并行 < 0.5s
XCTAssertLessThan(elapsed, 0.5, ...)
```

### MockURLProtocol 设计要点
- 静态 `handler: (URLRequest) async -> (HTTPURLResponse, Data?)` 支持 sleep 模拟延迟
- `Task.detached` 避免继承调用方 actor，确保 3 个并发请求真正并行
- `capturedRequests` 数组用于断言 URL / Header / Method

### 按 host 分流 mock
- `api.kimi.com` → Kimi 格式 JSON
- `api.minimax.io` → MiniMax 格式 JSON
- `api.z.ai` → GLM 格式 JSON（limits 数组）

## 六、踩坑记录（与 v0.1 v0.2 区别）

| 坑 | 解决 |
|----|------|
| `ActivityLevel` 缺 `CaseIterable` → `allCases` 编译失败 | 加 `CaseIterable` 协议 |
| `quota.primaryWindow?.used` (Double?) 推断不出 `Double` | 改 `XCTUnwrap(quota.primaryWindow)` |
| 自定义 `XCTAssertThrowsError<T, E: Error>` 泛型推断失败 | 删掉，用标准 XCTest 实现 |
| `var callCount` 在 closure 中并发修改警告 | 用 `actor TickCounter` 替代 |
| `Task { @MainActor in ... }` 让 3 个 sleep 串行化 | 改 `Task.detached` |
| GLM mock JSON 用了 Kimi 格式 → decodingFailed | 按 host 分流 mock 返回 3 平台真实格式 |
| `MockURLProtocol` 找不到 `HTTPClient` | 加 `@testable import QuotaMonitor` |
| `extraneous '}'` SmartRefreshSchedulerTests | Edit 时多粘贴了一个 `}`，删除 |
